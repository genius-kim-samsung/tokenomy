# Tokenomy

AI 코딩 토큰 지출을 추적하는 로컬 가계부. 이 글로서리는 프로젝트 고유의 도메인 용어만 정의한다(구현 세부는 CLAUDE.md / docs 참조).

## 사용 형태(usage mode)

사용자는 계정 성격에 따라 둘 중 하나의 형태로 앱을 쓴다. 형태는 공식 사용량 응답의 모양으로 판정한다.

**엔터프라이즈/API**:
공식 API가 provider-강제 **USD 한도**(`spend.limit`·달러 크레딧 창·Codex 크레딧 한도)를 노출하는 형태. 토큰/달러 단위로 실제 과금되며, 공식 한도가 곧 기간 예산이다.
_Avoid_: PAYG, 유료 플랜

**개인 구독제**:
정액제라 토큰당 과금이 없는 형태. 공식 API는 USD 한도 대신 rate-window(%)만 준다. 월 USD 예산은 이 형태에 의미가 없다.
_Avoid_: 무료, Pro/Max(특정 플랜명으로 부르지 말 것)

## 한도와 사용량

**공식 사용량**:
각 CLI의 로컬 OAuth 토큰으로 공식 API를 읽어 얻는, provider가 강제하는 실제 소진·한도·리셋. 한도/잔여의 **정본(source of truth)**. 범위는 **인증된 계정 한 명이 모든 기기에서 쓴 모든 표면**(Claude=Chat+Claude Code+Cowork)의 합 — 토큰 수가 아니라 구독 회계 단위(USD/크레딧/% of limit)로 보고된다. 엔터프라이즈 `spend`는 조직 공유 풀이 아니라 **그 사용자 개인 할당 소비**다(per-user 토큰으로 읽으므로).
_Avoid_: 예산, budget, 수동 한도 — "사용자가 정한 목표 한도"라는 옛 개념은 모델에 없다. **공식 "API" 사용량**(개발자 API 콘솔과 혼동 — 이건 OAuth 구독 사용량이다).

**로컬 사용량**:
각 CLI의 세션 로그(JSONL)를 파싱해 **이 기기에서** 집계한 사용량. 토큰 수는 정확히 측정하지만 USD는 정가(pricing.json) **환산 추정**이다. 공식이 못 주는 분해(프로젝트별·세션별·캐시 효율·시계열)를 제공하는 게 존재 이유다.
_Avoid_: 총 사용량·실제 비용(부분 표본이지 정본 총량이 아니다 — 아래 불변식 참조).

**공식 ⊇ 로컬 (불변식)**:
로컬 사용량은 공식 사용량의 **부분집합**이다 — 세 축에서 항상 좁다: ① 표면(로컬 Claude=Code만 / 공식=Chat+Code+Cowork) ② 기기(로컬=이 기기만 / 공식=전 기기) ③ 단위·근거(로컬=토큰×정가 추정 / 공식=구독 회계 정본). 따라서 **로컬은 절대 "총량"으로 제시하지 않는다** — 공식이 절대 위치·한도의 앵커, 로컬은 그 아래 깔리는 분해·속도 레이어다(위치=공식·기울기=로컬, ADR 0004의 일반화).
_Avoid_: 공식과 로컬을 동일 단위로 직접 비교(정가 추정 vs 구독 회계라 동일 비교 자체가 부정확).

**커버리지(coverage)**:
로컬 사용량이 공식 사용량의 몇 %를 설명하는가 = `로컬 USD(좌석 단가 환산) ÷ 공식 spend(개인 할당 소비)`. 이 기기 Code/Codex가 내 계정 전체 청구에서 차지하는 비중 — "내 로컬 장부가 실제 청구의 얼마를 담나"라는 자기 사각지대를 고백하는 값. **엔터프라이즈 모드에서만 성립**한다(개인 구독제는 공식이 % rate-window만 주어 분모가 없어 계산 불가 — 정성 단서로 대체). 본질상 **근사**다: 도구 사용 부가비(웹검색·코드실행)를 로컬이 못 잡고, 캐시 배수·좌석 단가가 pricing.json에 정확히·최신으로 들어가 있어야 성립한다. 헤드라인이 아니라 조용한 진단용.
_Avoid_: 정확도·정합률(근사임을 가리는 말), 개인 모드에서의 커버리지(성립 안 함 — "공식 ⊇ 로컬" 정성 단서로 대체).

**rate-window**:
개인 구독제에서 공식 API가 주는 회전 시간창(5시간·7일)의 이용률(%). 개인 구독제 사용자의 핵심 actionable 신호.
_Avoid_: 이용률 창(코드 라벨일 뿐, 도메인 용어로는 rate-window)

**사용량 전용 view**:
공식 데이터가 없을 때(공식 취득 off·미성공, 또는 한도를 안 주는 계정) 보여주는 폴백 화면. 로컬 JSONL에서 추정한 사용량 차트만 있고 한도/번다운은 없다.
_Avoid_: 추적 전용 모드(옛 "예산 0" 상태를 가리키던 말 — 폐기)

## 전망

"지금 추세대로 쓰면 이번 달 끝에 어떻게 되나"를 사람에게 보여주는 예측. 한도가 있는 형태에서만 의미가 있다(개인 구독제엔 한도가 없어 성립 안 함).

**통합 풀(combined pool)**:
공식 USD/크레딧 한도가 **있는** provider들을 하나로 합친 한도·소진·잔여. 여러 AI가 사실상 한 묶음의 작업 예산이라는 사용자 관점("Claude가 떨어지면 Codex로 일한다")을 반영한다. 한도를 안 주는 provider(개인 구독제 rate-window)는 풀에서 제외한다. 한 provider가 여러 USD 한도 버킷을 가지면(예: 월 사용 한도 + 포함 크레딧) 모두 합산한다 — 실제 소비되는 예산 전부가 풀이다(카드 게이지는 버킷별로 따로 보여주되, 풀은 합산값).
_Avoid_: 전체 합계(단순 총지출과 구분 — 풀은 한도 대비 잔여까지 포함한 개념).

**소비속도(burn rate)**:
한 영업일에 소진하는 양(USD/영업일). **오늘 포함 최근 N주(트레일링 창, 기본 2주)**의 로컬 소비 ÷ 그 창의 영업일로 추정한다(공식 스냅샷은 희소해 기울기 근거로 부적합). 청구 **리셋과 무관한 행동 속성**이라 리셋 경계를 가로질러 추정한다 — 위치(used)는 공식이라 리셋되지만, 기울기는 로컬·트레일링이라 리셋되지 않는다. 창 길이는 `forecast_settings.rate_window_weeks`(1주 단위)로 조정한다.
_Avoid_: 월초 누적 평균(리셋에 묶인 옛 정의 — 폐기), 일일 비용(burn rate는 비용 자체가 아니라 한도를 갉아먹는 속도라는 함의).

**공식 사용량 스냅샷 이력(official-snapshot history)**:
공식 API를 주기적으로 읽어 얻은 **공식 사용량** 스냅샷을 찍은 시각과 함께 누적 저장한 시계열. 한 시점만 보여주는 공식 사용량(정본)과 달리 "지나온 실제 궤적"을 만든다. 전망 차트에서 현재 위치 한 점을 실제 과거 실선으로 바꿔, 예상선(소비속도=로컬 기울기)이 관측된 공식 궤적 위에 얹히게 한다. **소진형 한도(누적 USD/크레딧)에만** 의미가 있다 — rate-window는 진동형이라 궤적이 "소진을 향해 다가감"을 못 보여줘 제외한다. 소비속도와 별개다(이건 공식 정본의 실제 과거 곡선, 소비속도는 로컬 추정 기울기). 수집은 상주 중 자동 갱신 간격마다 백그라운드로 일어난다(ADR 0007 — ADR 0006의 "숨김 중 주기작업 없음"을 공식 갱신에 한해 보완). 앱이 완전히 종료·절전된 구간엔 여전히 구멍이 생길 수 있으니 빈 구간을 "사용 0"으로 오인하지 말 것.
_Avoid_: 사용량 추이(로컬 로그 기반 추이와 혼동 — 이건 공식 정본 기반), 번다운(옛 수동 예산 곡선 — 폐기).

**전망(outlook)**:
통합 풀의 현재 위치(공식 used)에서 소비속도로 이번 달 끝(달력 월말 KST)까지 직선 연장했을 때의 예상 잔여. 사람에게는 **여유**(양수, 한도 안에 남음)나 **부족**(음수 → 소진 예상일)으로 보여준다. 전망은 본질상 추정이다(기울기가 로컬 추정).
_Avoid_: 번다운(옛 수동 예산 소진 곡선 — 폐기), 예산 잔여(예산 개념은 모델에 없다).

**여유 / 부족(surplus / shortfall)**:
전망의 부호. 여유 = 이 속도면 월말까지 한도가 남는다. 부족 = 월말 전에 한도를 넘긴다 → 그 도달 시점이 **소진 예상일**.
_Avoid_: 초과/미달(여유·부족으로 통일).

## 데이터 새로고침

화면 데이터를 최신으로 만드는 동작은 본질이 다른 둘로 갈린다. 한 단어 "새로고침"으로 뭉뚱그리지 않는다.

**수집**:
로컬에 쌓인 각 CLI의 세션 로그(JSONL)를 재스캔해 DB에 반영하는 동작. 로컬·즉시·항상 안전(네트워크 없음·실패 없음). 전 페이지 데이터에 영향.
_Avoid_: ingest를 "새로고침"으로만 부르지 말 것 — 갱신과 구분한다.

**갱신**:
공식 API를 호출해 **공식 사용량**(사용량·잔여·한도)을 다시 받아오는 동작. 아웃바운드 네트워크·실패 가능·quota를 CLI와 공유. "AI별 사용량" 카드에만 영향.
_Avoid_: 공식 fetch를 "새로고침"으로만 부르지 말 것 — 수집과 구분한다. "공식 갱신"처럼 "공식"을 덧붙일 필요도 없다(카드 문맥이 이미 공식임을 말해준다).

**자동 갱신 간격**:
창이 열려 있는 동안 갱신을 자동 폴링하는 주기이자, 자동 호출의 최소 간격(throttle 겸용). 수동 갱신은 이 간격을 무시한다(사용자의 명시적 의사 우선).
_Avoid_: throttle, min_interval — 방어적 의미가 아니라 "얼마나 자주 자동 갱신하나"라는 적극적 의미다.

## 활성 AI

**활성 AI** (코드: 활성 provider):
사용자가 앱에서 보기로 켠 provider의 집합. 공식 API 호출 대상이자, 대시보드·내역·차원별 등 **모든 화면의 표시 대상**을 게이트한다. 끈 provider는 데이터를 보존한 채 전 화면에서 숨겨진다(다시 켜면 복원). 화면의 "전체"는 곧 활성 AI의 합산을 뜻한다(DB 전체가 아니다).
_Avoid_: tracked providers(옛 이름 — 공식 취득만 게이트하던 시절의 협의), 표시 대상 AI

**provider**:
하나의 AI 도구/로그 출처(claude·codex 등). 새 도구 추가는 provider 하나를 더하는 것과 같다.
_Avoid_: tool, AI, 벤더(코드 전반에서 provider로 통일)

## 실행 수명주기(lifecycle)

앱이 "한 번 보고 끝"인지 "계속 켜두는" 것인지를 가르는 형태. webview 경로(네이티브 창)에서만 분기하고, 브라우저 fallback·개발 모드는 늘 단발이다.

**단발 모드(single-shot)**:
실행 시 수집 1회, 창을 닫으면 프로세스가 완전히 종료되는 형태. 브라우저 fallback과 개발 모드(직접 uvicorn)는 숨길 창이 없어 항상 이 모드다.
_Avoid_: 단순 실행(상주 모드와의 대비를 드러내는 용어).

**상주 모드(resident mode)**:
창을 닫아도 종료되지 않고 트레이에 머물러, 사용자가 사용량을 수시로 흘끗 볼 수 있게 계속 켜둔 형태. 종료는 트레이에서 명시적으로만 한다. webview 경로 전용.
_Avoid_: 백그라운드 모드(상주 중에도 주기 작업은 돌지 않는다 — 복원 시점에만 수집).

**트레이 숨김(hide to tray)**:
창 X 버튼이 종료가 아니라 창만 감추는 동작. 다시 보려면 트레이 아이콘을 더블클릭하거나 우클릭 메뉴의 "열기"를 쓴다. 숨김 상태에서 창을 복원할 때 수집 1회가 따라온다.
_Avoid_: 최소화(작업표시줄 최소화와 다르다 — 작업표시줄에서도 사라진다), 종료(트레이 숨김은 종료가 아니다).

**미니 뷰(mini view)**:
큰 창(대시보드)과 **한 번에 하나만 보이는 별도 창** — 항상 위(always-on-top) 프레임리스 미니 창이다. 큰 창과 **배타적으로 전환**된다(동시에 뜨지 않는다 — 한쪽을 띄우면 다른 쪽은 숨는다). 활성 AI별 현재 잔여/사용을 흘끗 보게 한다. 데이터는 **공식 스냅샷만** 읽어(수집/ingest와 무관 — 로컬 비용·프로젝트 집계는 큰 창의 몫) 켜진 동안 자동 갱신 간격으로 스스로 폴링한다. 일반뷰의 "미니뷰" 버튼과 미니뷰의 "일반뷰" 버튼으로 서로 오가며, **마지막 본 뷰**가 영속된다(트레이 "열기"·재실행은 마지막 뷰로 복원, 최초 실행은 일반뷰). 미니 창은 첫 전환 때 생성되고 이후 숨김/표시로 재사용된다. 양쪽 모두 창 닫기(일반뷰 X·미니뷰 ✕)는 트레이 숨김(파괴 아님)이다. 상주 모드의 webview 경로 전용(브라우저 fallback·개발 모드엔 없다).
_Avoid_: 미니 모드(이 글로서리에서 "모드"는 단발/상주 같은 수명주기 축이다 — 미니 뷰는 상주 모드 안에서 큰 창과 배타 전환되는 뷰이지 제3의 수명주기 모드가 아니다), 동반 창(큰 창과 동시에 뜨는 두 번째 창이 아니다 — 배타적으로 전환되어 한 번에 하나만 보인다), 위젯·HUD(클릭 통과/오버레이 함의는 모델에 없다 — 미니 뷰는 클릭 가능한 일반 창이다), 미니 대시보드(대시보드 축소판이 아니라 흘끗용 현재-상태 면).
