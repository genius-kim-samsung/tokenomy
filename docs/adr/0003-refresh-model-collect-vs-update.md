# 새로고침을 "수집"과 "갱신" 두 동작으로 가르고, 갱신만 자동 주기·수동은 throttle을 무시한다

앱에는 화면 데이터를 최신으로 만드는 동작이 둘 있다. **수집**(로컬 JSONL 재스캔 → DB)과 **갱신**(공식 API 호출 → 사용량/잔여)이다. 이 둘은 본질이 정반대다 — 수집은 로컬·즉시·항상 안전(실패 없음)이고, 갱신은 아웃바운드 네트워크·실패 가능·quota를 CLI와 공유한다. 그런데 코드·UI가 둘 다 "새로고침"이라는 한 단어와 같은 `↻` 아이콘으로 불러 세 가지 혼란을 낳았다. (1) 사이드바 "↻ 새로고침"이 사실 `cmd_ingest`의 데몬 스레드로 **갱신까지 몰래** 했고, (2) 자동 호출 빈도를 막으려던 5분 throttle이 **수동 클릭까지 막아** 사용자가 갱신 버튼을 눌러도 조용히 아무 일도 안 일어났고, (3) 신선도 표기가 비대칭이었다(사이드바는 절대시각, 게다가 "마지막 수집 시각"이 아니라 최신 메시지 ts였다 / 카드는 상대시간). 그래서 둘을 **명확히 구분되는 두 도메인 동작**으로 가르되 **일관된 UX 패턴**으로 묶기로 했다.

핵심 규칙:

- **버튼 위치 = 영향 범위.** 수집은 전 페이지 데이터를 바꾸니 **사이드바(전역)**, 갱신은 "AI별 사용량" 카드만 바꾸니 **그 섹션 안**(섹션 "전체 갱신" + 카드별 개별). 아이콘은 `↻`로 통일하되 **라벨("수집"/"갱신")로 구분**한다. 사용자가 위치로 "이 버튼이 뭘 바꾸나"를 학습한다.
- **수동은 각자 자기 일만, 자동(起動)은 둘 다.** 수동 수집 버튼은 네트워크를 건드리지 않고(→ "수집했는데 왜 네트워크 오류?"가 사라짐), 수동 갱신 버튼은 수집을 건드리지 않는다. 앱 起動 시에는 수집+갱신을 함께 돌려 첫 진입을 한 번에 최신화한다.
- **수동 갱신은 throttle을 무시한다.** throttle은 우리의 자동 호출 빈도만 제어할 의도였으므로, 사용자가 직접 누른 명시적 의사에는 적용하지 않는다. 실패해도 마지막 스냅샷이 유지돼 안전하다.
- **갱신만 자동 주기를 둔다.** rate-window·소진 카운트다운은 시간이 지나며 변하므로(창을 열어두고 "얼마 남았나" 보는 동안 의미가 있다) 창이 열린 동안 완만히 자동 폴링한다. 수집은 로컬 작업을 해야 변하고 보통 "작업 끝내고 확인"하는 흐름이라 자동 주기의 가치가 낮아 **수동(+起動)만** 둔다.
- **`min_interval_minutes`를 "자동 갱신 간격"으로 재정의.** 방어적 throttle 변수 하나가 곧 폴링 주기 겸 자동 호출 최소 간격이 된다(기본 **10분** — quota를 CLI와 공유하므로 보수적으로). 설정 라벨도 "자동 갱신 간격(분)"으로 바꾼다.
- **신선도 대칭.** 둘 다 "수집: N분 전" / "갱신: N분 전" 상대시간 + hover 절대시각으로 통일하고, 사이드바 신선도의 의미를 **마지막 수집 실행 시각**(`record_ingest`)으로 교정한다. "N분 전"이 정적으로 굳지 않도록 절대시각 `data-ts`를 심고 가벼운 JS가 표시만 tick한다(네트워크 무관).

## 고려한 대안

- **하나의 "새로고침"으로 통합(한 버튼이 다 처리, 내부 구분 숨김)**: 거부. 둘은 실패 양상·비용·안전성이 근본적으로 달라서, 사용자가 "왜 한쪽은 눌러도 안 되지?", "왜 한쪽만 실패하지?"를 이해하려면 다른 일임을 알아야 한다. 한 단어로 뭉뚱그리면 지금의 혼란이 그대로 남는다.
- **수동도 throttle 유지하되 "방금 갱신함" 피드백만 추가**: 거부. 조용한 무시는 고치지만, 사용자가 fresh를 원해 누른 의사를 여전히 거스른다. bypass하면 "눌러도 안 됨"이라는 모순 자체가 사라진다.
- **수집도 자동 주기**: 거부. tokenomy 창을 보면서 동시에 다른 창에서 코딩하는 경우는 드물어 자동 수집의 가치가 낮고, 의미 없는 로컬 재스캔만 반복된다. 두 동작에 다른 주기 정책을 주는 건 성격(시간 의존성) 차이로 정당화되며, 그 차이는 신선도 표기로 드러난다.
- **자동 주기 없이 起動 1회 + 수동만**: 진지하게 검토(단발 데스크톱 앱의 "스냅샷" 정신모델 유지·최소 복잡도). 갱신만 완만한 폴링으로 절충하되, 부분 교체로 "라이브 대시보드"화의 부작용(화면 튐)을 막는 선에서 받아들였다.

## 결과

- `cmd_ingest`의 데몬 갱신 훅을 떼어 수집을 순수화한다. CLI `ingest`(터미널)도 네트워크를 건드리지 않는 수집만 한다.
- 起動 자동 갱신은 별도 데몬 스레드 대신 **대시보드 로드 시 자동 폴링의 첫 tick**(`hx-trigger="load"`)이 겸한다 — 창이 뜨면 "/"가 로드되며 갱신이 1회 일어나고, 이후 '자동 갱신 간격'마다 폴링한다. `launcher`는 수집만 동기 실행하고 무변경(起動 스레드를 늘리지 않아 단순).
- tracked 전체 갱신 로직은 `official_fetch.refresh_tracked`로 모아 자동 폴링 라우트가 재사용한다.
- `fetch_provider`에 manual 경로를 두어 수동 트리거는 throttle 게이트를 건너뛴다. 자동(load·폴링)만 게이트를 거친다.
- 웹 경로: `POST /ingest`(수집만) · `POST /official/refresh`(수동 갱신, bypass) · 자동 폴링용 "AI별 사용량" 섹션 partial 라우트(throttle 적용, htmx 부분교체).
- 신선도 헬퍼·템플릿·설정 라벨·상대시간 tick JS가 이 모델을 따라 갱신된다.
